External Pentest, From Domain to Report.

Berckley is an external attack-surface scanner with a built-in validation layer that filters false positives, classifies every host by ownership, ranks findings by contextual risk, and ships executive + technical HTML reports. This guide walks you from a fresh install to your first signed-off report — assume nothing.

Launching a Scan

Detailed flow with all the knobs.

The New Scan modal

Click + New Scan. You'll get a form with:

Domains — One or more comma-separated root domains (example.com, sub.example.com). The scanner will independently enumerate subdomains during recon, so providing just the apex is usually enough.

IP Targets — Optional. Comma-separated IP[:port] entries for when you also want raw IPs scanned (e.g. 1.2.3.4, 5.6.7.8:8443).

Phase — Leave on All for a full scan. Pick a single number if you want to re-run only a part. Phases are:

  • 1 OSINT (passive intel: WHOIS, GitHub orgs, Azure AD tenants, paste leaks)
  • 2 DNS (records, AXFR, DKIM selectors, MTA-STS, SPF, DMARC, takeover fingerprints)
  • 3 Recon (subdomain enum, masscan, vhosts, IIS tilde, WAF detect, sensitive files)
  • 4 TLS (cert chain, ciphers, OCSP, hostname match)
  • 5 Headers (HSTS, CSP, X-Frame, X-Content-Type, etc.)
  • 6 WebApp (SAP, VPN portals, cache poisoning, katana crawl, arjun, SSTI, Firebase, lookalikes, S3/Azure/GCS buckets, JS secret mining, Docker/K8s exposure, NoSQL/cache without auth)
  • 7 CVE (nuclei against the live host list)
  • 8 Auth (admin panels, default creds)
  • 9 Report (consolidates the findings TSV)

Threads / Rate — Concurrency and req/s limit. Defaults (20 / 50) are conservative. Bump rate to 100 if you own the target and don't worry about WAF blocking.

Tag — Optional. Appended to the scan directory name and shown in the sidebar (e.g. "Scan Report · 2026-05-25 14:30 · prod-audit").

Optional: Discover Scope first

Before launching, click 🔍 Discover Scope. The dashboard runs a fast passive pre-flight (DNS + crt.sh + lightweight CNAME classification) and shows a collapsible tree of every subdomain it found, grouped by root, with the ownership class pre-labeled. Defaults are sensible:

  • Owned and SaaS ✓ checked by default
  • Cdn, Internal, External ✗ unchecked (provider-managed or not yours)
  • Hosts that don't resolve are unchecked + struck through

Tweak the boxes, add custom subdomains via the + add input under each root, then click ▶ Launch. The dashboard writes your curated list to scope.txt inside the scan directory and the scanner uses that as input.

Tip. If crt.sh is unreachable (it's frequently flaky), the dashboard falls back to subdomain lists from previous scans of the same root on the host. First-time scans against a new domain might return no discovery — that's fine, launch with just the root and the scanner's own enum will fill the gap.

While the scan runs

The dashboard switches you to Live Log with an active SSE stream of the scanner's master.log. Color codes:

  • [CRITICAL] / [HIGH] / [MEDIUM] / [LOW] — finding emitted
  • [+] ok step / [*] info / [>] sub-step / [!] warning / [-] error
Don't rebuild the Docker container while a scan is running. The scanner runs as a child of the dashboard process; rebuilding kills it mid-flight. Findings already written to findings.tsv survive, but the scan will be incomplete.

Tabs Reference

What each tab shows, when you'd open it, and what to look for.

Overview
Open first, always.

Posture at a glance. A headline Security Posture Score (0–100 with an A–F grade), four big severity tiles, a banner showing validation deltas ("raw 173 → 162, −11"), a Findings by Security Domain card row, and a Top Hosts by Risk table sorted by the contextual scoring formula severity × exploitability × ownership × host criticality.

The score is 100 − (45·C + 18·H + 6·M + 1.5·L) over the validated findings (100 = clean); an open CRITICAL caps the grade at D, an open HIGH at B. Click any domain card to jump to the Findings tab filtered by that domain.

The little ▶ Run button next to "Validation" triggers the false-positive filter + ownership classification + persistent suppression list. Run it before doing anything else with the scan results.

Findings
Day-to-day analyst view.

Every issue the scanner produced, optionally filtered. Columns: severity, risk score (color-coded), category, scope, owner tag, description, and a ✕ button to suppress as false positive.

Toolbar offers: full-text filter, severity chips, ownership chips, security-domain chips (Network, Crypto/TLS, Web App, Email/DNS, Cloud, Secrets, Access Control — each finding also shows a colored domain badge), a Show Raw (Pre-Validation) toggle to compare against unfiltered scanner output, and Sort by Risk to put the most actionable items at the top.

Validation
When you want to know why.

Audit trail for every rule decision: which findings were suppressed as confirmed false positives, which were downgraded because the host is on third-party infra, and why. Filter by verdict, see the rule name + reason per row.

Rules in order: user-suppression (your allowlist), rfc1918-not-external, snmp-reverify, email-auth-no-mx, cert-mismatch-cname-to-saas, ownership-infra.

Ownership
"Whose infra is this?"

Every host classified as Owned, Saas, Cdn, Cloud_shared, Internal, External, or Unknown — with provider (Microsoft, Cloudflare, etc.), IP, ASN, and the evidence the classifier used (CNAME suffix, WHOIS Org, ASN match, or root-domain match).

Use the class chips to filter. Findings on third-party infra are automatically downgraded by the validation rule, but this tab shows you the full per-host picture.

Suppressions
After the first review pass.

The persistent allowlist of accepted false positives. Entries survive re-runs and re-validation; the file lives on disk at $PENTEST_ROOT/suppressions.json.

Add entries by clicking the ✕ on any finding row — the modal pre-fills category and scope. Edit scope to use a glob (e.g. *.example.com or 10.20.*) to match many findings with one entry. Reason is required for audit. Expires is optional ISO date (e.g. 2026-12-31) for time-boxed acceptance.

Diff
Recurring engagements.

Compare two runs. Auto-picks the most recent prior scan with overlapping input domains; pick manually with the dropdown if you want a specific baseline.

Buckets: New (regressions / fresh exposure), Fixed (closed since previous), Changed (severity moved), Unchanged. Severity delta cards at the top show net movement per severity tier (e.g. CRITICAL −11, HIGH −19).

Assets
Recon deep-dive.

Raw inventory of everything the scanner discovered, grouped by type: SUBDOMAINS, ASN, CRAWL (URLs found), GITHUB (public repos in the org), AZURE_AD (M365 tenant IDs), FEDERATED_AUTH (SSO endpoints), TECH_VERSION, and more.

Use the type chips to drill into one category at a time. Good for understanding the scope of what you're testing.

Extract
Handing data to other tools.

Export the identified host list to feed other tools: TXT (one host per line — drop into nmap -iL, nuclei -l, ffuf -w, httpx -l), CSV (with ownership + findings count + max severity), or JSON (structured for scripts and dashboards).

Filter by ownership class (e.g. only Owned), toggle Only Hosts With Findings to skip empty rows. Buttons: ⧉ Copy (clipboard) or ↓ Download.

Live Log
During an active scan.

Streams the scanner's master.log in real time over SSE. Colorized by line prefix. Use ▶ Attach Stream if it disconnected, ■ Detach to pause output, and ⛔ Stop Scan to SIGTERM the running scanner (everything written to disk so far is preserved).

Reports
When you're ready to share.

Two HTML reports, both built from validated data:

  • Management Report — executive layout (serif navy consultancy look), severity summary, ownership breakdown, top risks with quick wins, change-since-previous-engagement section.
  • SOC / Technical Report — modern hacker layout matching the dashboard, detailed finding cards with evidence + remediation, raw TSV preserved at the end for audit.

⚙ Generate rebuilds the HTML from current data (re-run any time after re-validation). ↗ Open opens it in a new tab; the file downloads with a friendly date-based name (e.g. Scan_Report_2026-05-17_example_soc.html).

Recommended Workflow

The path that minimizes noise and produces a defensible report.

  1. Launch the scan with full phases (All). Walk away — 30 to 90 min.
  2. Run Validation from the Overview tab. ~15–30 s. This step:
    • Re-probes SNMP findings (drops scan-time false positives)
    • Classifies every host's ownership via WHOIS + CNAME suffix matching
    • Applies your persistent suppression list (suppressions.json)
    • Downgrades infra findings on third-party SaaS / CDN / cloud-shared
    • Writes findings_validated.tsv + findings_audit.tsv + ownership.tsv
  3. Review the Findings tab sorted by risk. Top 10 hosts on Overview point you at the highest-value items.
  4. Triage: click ✕ on confirmed false positives → write the reason → they vanish from the validated view and persist in suppressions.json for future runs.
  5. Run Validation again so your new suppressions take effect.
  6. Generate the Reports. Send the Management one to the customer and keep the SOC one for the analyst handoff.
  7. (Next engagement) The Diff tab will compare automatically — focus attention on the New bucket.

Generating Reports

Two outputs, same data, different audiences.

Management Report

For: CISO, CTO, board, customer execs.

  • Consultancy-style serif navy theme
  • Executive summary with finding counts
  • Ownership breakdown (owned vs third-party)
  • Top risks + quick wins
  • Change-since-previous-engagement section
  • No raw evidence — clean narrative

SOC / Technical Report

For: SOC analysts, blue team, IT.

  • Modern hacker / terminal theme (matches dashboard)
  • Detailed finding card per category
  • Affected scopes as severity-colored pills
  • Per-category verify command + remediation guidance
  • Raw findings TSV preserved at the end for audit

Both reports:

  • Read findings_validated.tsv when present (fall back to raw findings.tsv)
  • Include 4 injected sections: Validation Summary, Asset Ownership Breakdown, Risk-Weighted View, Change Since Previous Engagement. Each section auto-skips if its data isn't available.
  • Download with date-based filename (Scan_Report_2026-05-17_example_management.html).

How To…

Common tasks, short answers.

Suppress a false positive

  1. Open Findings, locate the finding row.
  2. Click the ✕ button at the end of the row.
  3. Modal pre-fills category and scope. Edit scope to a glob if you want it to match many findings (*.example.com, 10.20.*, ? for single-char wildcard).
  4. Write a reason (required for audit). Optionally set an expiry date.
  5. Click ✕ Add Suppression, then go to Overview and click ▶ Run on Validation to re-apply.

Filter findings by ownership

On the Findings tab, click any ownership chip (Owned, Saas, etc.) to narrow the table. Multi-select supported — clicking another chip ANDs them. To see findings on infrastructure you actually control, click only Owned.

Export hosts to feed other tools

  1. Open Extract.
  2. Pick ownership classes (e.g. Owned only).
  3. Toggle Only Hosts With Findings if you want to skip empty hosts.
  4. Pick a format: TXT for tool input, CSV for spreadsheets, JSON for scripts.
  5. Click ⧉ Copy or ↓ Download.
# Example: feed Berckley's TXT export to nuclei
curl -s 'http://127.0.0.1:8080/api/scans/<scan-name>/extract?format=txt&owner_class=OWNED' \
  > targets.txt
nuclei -l targets.txt -severity high,critical

Compare against a previous scan

Open Diff. Leave the dropdown on auto for the most recent prior scan with overlapping domains, or pick one manually. Click ▶ Run Diff. Read the severity-delta cards first (e.g. HIGH −19 = good), then drill into the New bucket for fresh exposure.

Re-run only one phase

In the New Scan modal, set Phase to a single number (e.g. 6 for WebApp). Useful when you've added a new check and want to test against an existing recon dataset without re-doing the slow parts.

Stop a runaway scan

Open the running scan, go to Live Log, click ⛔ Stop Scan. Everything written to disk is preserved (findings, assets, recon files).

Troubleshooting

The most common landmines and how to step around them.

"Launch failed: 500 Internal Server Error"

Older builds had an asyncio bug in the launch endpoint. Update to the latest image (docker compose up --build -d) — the launch endpoint is now async and uses asyncio.create_task.

Note: even when the API returned 500, the scan often started anyway (Popen fired before the error). Check the sidebar — if you see a recent pentest_* entry, that scan is running. Stop it via Live Log if you don't want it.

The Discover Scope returns 0 subdomains

crt.sh is frequently flaky (502, timeout). The pre-flight retries 3 times then falls back to subdomain data from prior scans of the same root. First-time scans against a new domain may show 0 — just launch with the root only and the scanner's own enum will find them during phase 3.

The Live Log stream disconnects

SSE timeouts behind proxies are common. Click ▶ Attach Stream to reconnect. The full log is always on disk at <scan-dir>/report/master.log if you'd rather tail -f it.

Validation takes ~20s

Normal. It does parallel whois + DNS + (optionally) snmpget probes for every unique host in the findings TSV. The result is cached in report/ownership.tsv, so subsequent reads are instant — only re-runs re-probe.

I rebuilt the container and lost a running scan

Expected. The scanner is a child process of the dashboard container; rebuild = SIGTERM. Findings already written to findings.tsv survive (the file is on the host-mounted volume). To resume, re-run a single phase via the Phase selector to fill the gap.

Old scan directories accumulate

They're plain dirs under $PENTEST_ROOT (the repo root by default). rm -rf pentest_xyz to clean. If the container ran as root, you may need sudo.

Glossary

Vocabulary that shows up across the UI and reports.

Finding — A single issue the scanner identified, with severity, category, scope (host/URL/IP), and description. Stored as one tab-separated line in findings.tsv.

Scope — The host, URL, IP, or port this finding applies to.

Validation — Post-scan pass that filters false positives and re-weights findings. Produces findings_validated.tsv + findings_audit.tsv.

Ownership class — Who controls the host's infrastructure. Owned = you, Saas = SaaS tenant (M365, Heroku…), Cdn = edge, Cloud_shared = shared cloud IP, Internal = RFC1918, External = third-party domain, Unknown = couldn't decide.

Suppression — A persistent allowlist entry that hides a finding from the validated view across runs. Lives in suppressions.json.

Risk score — severity × exploitability × ownership × host_criticality. Surfaces the items worth fixing first.

Security domain — The one top-level category a finding belongs to: Network & Infrastructure, Cryptography & TLS, Web Application Security, Email & DNS Security, Cloud & Storage Exposure, Secrets & Info Disclosure, or Access Control & Auth. Single membership, deterministic. Defined in taxonomy.py; used for the Overview cards, Findings filter, and reports.

Posture score / grade — A 0–100 report-card score for the whole scan (100 = clean), with a letter grade: A 90+, B 80+, C 70+, D 60+, F <60. Computed as 100 − (45·C + 18·H + 6·M + 1.5·L) over validated findings, with severity ceilings (open CRITICAL ⇒ max D, open HIGH ⇒ max B). Defined in scorecard.py.

Live host — A host that responded to HTTP/HTTPS during recon. Listed in recon/live_hosts.txt.

Scan directory — Per-scan output folder named pentest_<tag>_YYYYMMDD_HHMMSS, with subdirs for each phase (osint/, dns/, recon/, tls/, headers/, webapp/, cve/, auth/, cloud/, report/, assets/).
